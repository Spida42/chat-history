#### 3. 类与面向对象（关键差异点）

```csharp name=类与面向对象.cs
// C#的类定义（对比C++）
public class PLCDevice
{
    // ===== 字段（Field）：类似C++的成员变量 =====
    private string _ipAddress;
    private int _port;
    private bool _isConnected;

    // ===== 属性（Property）：C#独有！面试必问！=====
    // 属性 = 字段 + getter/setter 的语法糖
    // 比C++的getXxx()/setXxx()优雅得多
    public string IpAddress
    {
        get { return _ipAddress; }
        set { _ipAddress = value; }  // value是C#关键字，代表传入的值
    }

    // 自动属性（最常用的���写）
    public int Port { get; set; }
    public bool IsConnected { get; private set; }  // 外部只读，内部可写

    // ===== 构造函数（和C++一样）=====
    public PLCDevice(string ip, int port)
    {
        _ipAddress = ip;
        Port = port;
        IsConnected = false;
    }

    // ===== 方法 =====
    public void Connect()
    {
        // 连接PLC的逻辑
        IsConnected = true;
        Console.WriteLine($"已连接到 {_ipAddress}:{Port}");
        // 注意：$"..." 是字符串插值，类似Python的f"..."
    }
}

// 使用：
var plc = new PLCDevice("192.168.1.100", 502);
plc.Connect();
```

#### 4. 接口（Interface）— 面试高频考点

```csharp name=接口示例.cs
// 接口：定义"契约"，规定类必须实现哪些方法
// 工控场景：不同品牌的PLC（西门子、三菱、欧姆龙）都要实现统一的通信接口

public interface ICommunication
{
    bool Connect();
    bool Disconnect();
    byte[] SendAndReceive(byte[] data);
    bool IsConnected { get; }
}

// 西门子PLC通信类
public class SiemensPLC : ICommunication
{
    public bool IsConnected { get; private set; }
    
    public bool Connect()
    {
        // 西门子特有的连接逻辑（S7协议）
        IsConnected = true;
        return true;
    }
    
    public bool Disconnect()
    {
        IsConnected = false;
        return true;
    }
    
    public byte[] SendAndReceive(byte[] data)
    {
        // 发送并接收数据
        return new byte[0];
    }
}

// 三菱PLC通信类
public class MitsubishiPLC : ICommunication
{
    public bool IsConnected { get; private set; }
    
    public bool Connect()
    {
        // 三菱特有的连接逻辑（MC协议）
        IsConnected = true;
        return true;
    }
    
    // ... 实现其他方法
}

// 好处：上层代码不关心具体是什么PLC
// ICommunication plc = new SiemensPLC();  // 或 new MitsubishiPLC();
// plc.Connect();  // 多态！
```

## 委托与事件 — 上位机开发的灵魂

### 先从一个场景开始

```
假设你在开发一个上位机软件，要做这件事：

  PLC检测到温度超过100℃ 
  → 上位机收到这个报警信号
  → 界面上弹出红色警告
  → 同时把报警记录存到数据库
  → 同时发一条短信给工程师

问题来了：
  "收到报警信号"这个动作发生之后，
  要同时触发三件事，怎么做？
```

你可能会想：写三个 if 判断不就行了？可以，但很笨。**委托和事件就是解决这个问题的优雅方案。**

---

### 第一步：先理解"委托"

**委托 = 把一件事"委托"给别人去做**

生活类比：

```
你是老板（主程序）

你不可能所有事都自己做，所以你：
  → 把"打扫卫生"委托给保洁阿姨
  → 把"送快递"委托给快递小哥
  → 把"做饭"委托给厨师

"委托"就是：
  我不关心你是谁，
  我只关心你能不能完成我指定的这件事。
```

在代码里：

```
委托 = 一个"任务槽位"

你先定义一个槽位："我需要有人能处理报警"
然后把具体的处理方法"塞进去"
当报警发生时，槽位里的方法就会自动执行
```

最简单的代码：

```csharp name=委托基础.cs
// 第1步：定义一个委托类型（定义"槽位的形状"）
// 意思是：谁要接这个任务，必须是"接收一个string参数、不返回值"的方法
delegate void AlarmHandler(string message);

// 第2步：写两个具体的处理方法
void 弹窗报警(string message)
{
    // 在界面上显示红色警告
}

void 存数据库(string message)
{
    // 把报警信息写入数据库
}

// 第3步：创建委托，把方法"塞进去"
AlarmHandler handler = 弹窗报警;   // 先塞一个
handler += 存数据库;               // 再塞一个（用 += 追加）

// 第4步：触发！两个方法都会执行
handler("温度超过100℃！");
// → 自动执行 弹窗报警("温度超过100℃！")
// → 自动执行 存数据库("温度超过100℃！")
```

重点理解这个流程：

```
handler  ← 这是一个"槽位"

往槽位里塞了两个方法：
  槽位 → [弹窗报警] [存数据库]

调用 handler("温度超过100℃！") 时：
  → 槽位里的方法一个接一个全部执行
  → 弹窗报警("温度超过100℃！")  ✅ 执行
  → 存数据库("温度超过100℃！")  ✅ 执行
```

---

### 第二步：再理解"事件"

**事件 = 对委托加了一层保护**

类比：

```
委托就像一个公告栏，任何人都能：
  → 往上面贴通知  （+=）
  → 撕掉通知      （-=）
  → 甚至直接清空整个公告栏  ← 这很危险！

事件就是给公告栏加了个玻璃罩：
  → 你可以往上面贴通知    （+= ✅）
  → 你可以撕掉自己的通知  （-= ✅）
  → 但你不能清空别人的通知（直接赋值 ❌ 不允许）
  → 也不能从外部随便触发  （只有公告栏的管理者能触发）
```

代码对比：

```csharp name=委托vs事件.cs
// ===== 用委托（没保护） =====
public AlarmHandler OnAlarm;

// 外部可以随便搞：
OnAlarm = null;              // 把所有处理方法全清空了！危险！
OnAlarm("假的报警");          // 从外部随便触发！危险！


// ===== 用事件（有保护） =====
public event AlarmHandler OnAlarm;

// 外部只能：
OnAlarm += 弹窗报警;          // ✅ 可以：订阅（添加处理方法）
OnAlarm -= 弹窗报警;          // ✅ 可以：取消订阅
OnAlarm = null;              // ❌ 不行：不能清空
OnAlarm("假的报警");          // ❌ 不行：不能从外部触发
```

> 就多了一个 **`event`** 关键字，就安全了。

---

### 第三步：上位机中怎么用？

完整的场景串起来：

```
上位机运行中：

  通信线程（后台默默工作）
    ↓
  每隔1秒从PLC读一次温度数据
    ↓
  发现温度 > 100℃
    ↓
  触发"报警事件"
    ↓ 事件触发后，所有订阅者自动执行：
  ├→ 界面模块：弹出红色警告框
  ├→ 数据库模块：把报警信息存起来
  └→ 通知模块：发短信给工程师
```

用代码描述就是：

```csharp name=上位机报警事件.cs
// ===== 通信类：负责和PLC通信 =====
class PLCCommunication
{
    // 定义一个报警事件（槽位）
    public event AlarmHandler OnAlarm;

    // 这个方法每隔1秒执行一次，读PLC数据
    public void 读取温度()
    {
        double 温度 = 从PLC读取();

        if (温度 > 100)
        {
            // 触发事件！通知所有订阅者
            OnAlarm?.Invoke("温度超过100℃！");
            // 这个 ? 的意思是：如果有人订阅了，才触发
        }
    }
}

// ===== 主程序：启动时把各模块"挂上去" =====
var plc = new PLCCommunication();

plc.OnAlarm += 弹窗报警;     // 界面模块订阅了这个事件
plc.OnAlarm += 存数据库;     // 数据库模块订阅了这个事件
plc.OnAlarm += 发短信;       // 通知模块订阅了这个事件

// 之后plc检测到报警时，三个方法全部自动执行！
```

### 总结：三句话记住

```
委托 = 一个可以装方法的"槽位"，能装多个方法
事件 = 委托 + 保护（外部只能订阅/取消，不能随意触发和清空）
上位机中的用法 = 当PLC发生了某个情况 → 自动通知所有相关模块去处理
```

> **面试话术：** "在我的上位机项目中，我使用事件机制来实现模块间的解耦。比如当通信模块检测到设备报警时，会触发报警事件，界面模块、数据库模块、通知模块分别订阅了这个事件，各自独立处理，互不影响。这样新增处理逻辑时只需要新增一个订阅者，不需要修改通信模块的代码。"

## 问题1：委托是"自动触发"的吗？

**不是自动触发的！是你手动喊它执行的。**

你理解反了一个地方：

```
handler("温度超过100℃！");

❌ 你的理解：括号里是"触发条件"，满足条件就自动执行
✅ 正确理解：括号里是"传给那些方法的参数"，这行代码本身就是在手动触发
```

类比：

```
你是班长，手里有个通讯录（handler），上面记了两个人：
  → 张三（弹窗报警）
  → 李四（存数据库）

handler("温度超过100℃！")  的意思是：

  你拿起电话，按照通讯录，
  挨个打电话告诉他们"温度超过100℃！"
  
  "温度超过100℃！"不是触发条件，
  而是你打电话时说的话（传递的消息）。

  至于什么时候打这个电话？
  是你（程序）自己决定的！比如在 if(温度>100) 里面打。
```

完整流程再看一次：

```
if (温度 > 100)                    ← 这里才是条件判断！
{
    handler("温度超过100℃！");      ← 条件满足了，你才手动触发handler
}

所以：
  条件判断 = if 语句负责
  触发执行 = 你手动调用 handler() 负责
  handler 本身不会自己判断、自己触发
```

## 问题2：`AlarmHandler` 是关键字吗？

**不是关键字！是你自己取的名字。**

```
delegate void AlarmHandler(string message);
         ↑        ↑              ↑
       关键字   你取的名字     参数要求

delegate → 这个才是C#的关键字，意思是"我要定义一个委托类型"
AlarmHandler → 这是你自己起的名字，想叫啥都行
```

就像：

```
class PLCDevice { }
  ↑      ↑
关键字  你取的名字

int count = 10;
 ↑    ↑
关键字 你取的名字

delegate void AlarmHandler(string message);
    ↑              ↑
  关键字         你取的名字（可以随便改）
```

下面这些写法都是合法的，效果一模一样：

```csharp name=委托命名.cs
// 这三种写法功能完全一样，只是名字不同
delegate void AlarmHandler(string message);      // 叫"报警处理器"
delegate void MyCallback(string message);        // 叫"我的回调"
delegate void 随便叫啥都行(string message);       // 甚至可以用中文（虽然没人这么干）
```

## 问题3：什么叫"从外部触发"？

**"外部"就是"这个类以外的地方"。**

```
你家（类的内部） vs 你家门外（类的外部）

┌─────────────────────────────┐
│  PLCCommunication 类（你家） │
│                             │
│  event OnAlarm              │
│                             │
│  读取温度()                  │
│  {                          │
│     if (温度 > 100)          │
│       OnAlarm("报警！");     │  ← 在家里触发，✅ 允许
│  }                          │
│                             │
└─────────────────────────────┘

┌─────────────────────────────┐
│  主程序（你家门外）           │
│                             │
│  plc.OnAlarm("假报警！");    │  ← 在外面触发，❌ 事件不允许！
│                             │
│  如果允许的话，任何人都能     │
│  随便制造假报警，系统就乱了   │
└─────────────────────────────┘
```

> 加了 `event` 关键字 = 只有类自己内部可以触发，外面的人只能"订阅"（等着接通知），不能"冒充"触发。

## 问题4：没有事件机制为什么就要改通信模块？

这是最好的问题！我对比给你看：

```
❌ 没有事件的写法：
────────────────
通信模块里要这样写：

读取温度()
{
    if (温度 > 100)
    {
        弹窗报警("温度超标！");       // 通信模块直接调用界面的方法
        存数据库("温度超标！");       // 通信模块直接调用数据库的方法
        发短信("温度超标！");         // 通信模块直接调用通知的方法
    }
}

现在老板说：加一个新功能——报警时要"发微信通知"
你就得打开通信模块的代码，加一行：

读取温度()
{
    if (温度 > 100)
    {
        弹窗报警("温度超标！");
        存数据库("温度超标！");
        发短信("温度超标！");
        发微信("温度超标！");         // ← 改了通信模块的代码！
    }
}

每次加功能都要改通信模块，很烦，而且容易改出bug
```

```
✅ 有事件的写法：
────────────────
通信模块里只写这些，永远不用改：

读取温度()
{
    if (温度 > 100)
    {
        OnAlarm("温度超标！");   // 只管发通知，不管谁在听
    }
}

要加"发微信"功能？
在主程序那边多加一行订阅就行：

plc.OnAlarm += 发微信;    // 在外面加，通信模块一个字都不用动！
```

对比一下：

```
                  没有事件          有事件
加新功能时      要改通信模块       不用改通信模块
                                   在外面 += 一下就行

这就是"解耦"的意思：
  通信模块不需要知道"谁在听"
  听的人也不需要知道"通信模块内部怎么工作"
  两边互不影响，各管各的
```

> **面试话术（更新版）：** "使用事件机制的好处是：通信模块只负责检测和发出通知，不需要知道有多少个模块在处理这个通知。新增功能时只需要订阅事件，不需要修改通信模块的代码，降低了模块之间的耦合度。"

## 多线程 — 上位机开发的必备技能

### 先理解一个问题：为什么需要多线程？

```
想象你是一个餐厅的唯一服务员（单线程）：

  顾客A说：帮我点餐
  你去点餐...
  
  这时候顾客B说：帮我倒水
  但你在忙，顾客B只能等...
  
  顾客C说：帮我结账
  继续等...

所有人排队等你一个人，效率很低！

现在餐厅请了三个服务员（多线程）：
  服务员1 → 专门负责点餐
  服务员2 → 专门负责倒水
  服务员3 → 专门负责结账
  
  三件事同时进行，谁都不用等！
```

上位机软件也是一样的：

```
上位机软件要同时做好几件事：

  任务1：界面要流畅响应（操作员点按钮不能卡死）
  任务2：每隔1秒从PLC读数据
  任务3：把数据存到数据库
  任务4：实时刷新界面上的温度、压力数值

如果只有一个线程：
  读PLC数据的时候 → 界面卡住不动！
  存数据库的时候 → 界面又卡住了！
  操作员以为软件崩了，直接投诉你。

所以必须用多线程：
  主线程 → 专门管界面（保证不卡）
  子线程1 → 后台默默读PLC数据
  子线程2 → 后台默默存数据库
```

这就是上位机开发中多线程的核心意义：

```
主线程 = 前台服务员（面对客户，必须随时响应）
子线程 = 后厨厨师（在后面干活，客户看不到）

绝对不能让厨师跑到前台去，
也不能让服务员去后厨炒菜，
各干各的，互不耽误。
```

### 怎么创建一个子线程？

最基本的方式：

```csharp name=创建线程.cs
// 主程序（主线程在运行）

// 创建一个子线程，告诉它去执行"读取PLC数据"这个方法
Thread 通信线程 = new Thread(读取PLC数据);

// 启动子线程（它会在后台默默运行）
通信线程.Start();

// 主线程继续管界面，两边互不影响

// ===== 这是子线程要执行的方法 =====
void 读取PLC数据()
{
    while (true)  // 一直循环
    {
        // 从PLC读温度
        // 从PLC读压力
        Thread.Sleep(1000);  // 休息1秒，然后继续读
    }
}
```

流程是这样的：

```
时间轴 →→→→→→→→→→→→→→→→→→→→→→→→→→→

主线程：  [管界面] [响应按钮] [管界面] [响应按钮] ...
                ↘
子线程：        [读PLC] [休息1秒] [读PLC] [休息1秒] ...

两条线同时在跑，互不干扰！
```

### ⚠️ 多线程最大的坑：抢资源

这是面试最爱问的点！

```
两个服务员（线程）同时要用同一个咖啡机（共享资源）：

  服务员A：我要用咖啡机做拿铁
  服务员B：我要用咖��机做美式

  如果两人同时操作，咖啡机就乱了！
  可能做出一杯"拿铁美式混合物"

这就叫"线程安全问题"
```

上位机中的真实场景：

```
子线程：刚从PLC读到温度 = 36.5，正在写入变量...
主线程：这时候来读这个变量，想显示在界面上...

如果同时读写同一个变量，可能读到"写了一半"的数据！
比如温度应该是36.5，你读到了36（后面的.5还没写完）
```

---

### 解决方案：加锁（lock）

```
加锁 = 给咖啡机装一个门，一次只能进一个人

服务员A进去了 → 把门锁上
服务员B来了 → 发现门锁着 → 在外面等
服务员A做完了 → 开门出来
服务员B进去 → 锁门 → 开始做咖啡
```

代码：

```csharp name=线程加锁.cs
// 这是一把"锁"（任意一个object就行）
private object 锁 = new object();

// 共享的数据
private double 当前温度 = 0;

// ===== 子线程：写数据 =====
void 读取PLC数据()
{
    while (true)
    {
        double 新温度 = 从PLC读取温度();

        lock (锁)                    // 进门，锁上
        {
            当前温度 = 新温度;         // 安全地写入
        }                            // 出门，自动开锁

        Thread.Sleep(1000);
    }
}

// ===== 主线程：读数据 =====
void 刷新界面()
{
    double 显示温度;

    lock (锁)                        // 进门，锁上
    {
        显示温度 = 当前温度;           // 安全地读取
    }                                // 出门，自动开锁

    界面标签.Text = 显示温度.ToString();
}
```

核心就一句话：

```
lock = 同一时间只有一个线程能进入这个花括号
       其他线程必须等它出来才能进去
```

### ⚠️ 多线程第二大坑：子线程不能直接动界面

这也是面试高频考点！

```
C#有一个规矩：
  界面上的控件（按钮、文本框等）只有主线程能碰！
  子线程不能直接修改界面！

为什么？
  就像一个画板，两个人同时画画会画乱。
  所以C#规定：只有一个人（主线程）能画。
```

那子线程想更新界面怎么办？

```
子线程不能自己去改界面，
但可以"拜托"主线程去改。

就像你在后厨（子线程），
想通知前台服务员（主线程）换一下菜单牌，
你得喊一声："服务员，帮我把今日特价改成红烧肉！"
而不是自己冲到前台去改。
```

代码：

```csharp name=跨线程更新界面.cs
// 子线程里想更新界面上的温度显示

// ❌ 直接改 → 程序会报错！
温度标签.Text = "36.5℃";  // 报错：不能从子线程访问界面控件！

// ✅ 拜托主线程去改 → 用 Invoke
this.Invoke(new Action(() =>
{
    温度标签.Text = "36.5℃";  // 主线程来执行，安全！
}));
```

你不需要完全理解 `Invoke` 的语法细节，只需要记住：

```
子线程想改界面 → 用 Invoke 包一层 → 让主线程代劳
这是固定套路，面试说出来就行
```

---

### 总结：多线程三个必记要点

```
1. 为什么用多线程？
   → 上位机要同时做多件事（界面 + 通信 + 存数据库）
   → 不用多线程界面会卡死

2. 线程安全问题怎么解决？
   → 多个线程访问同一个数据时，用 lock 加锁
   → 保证同一时间只有一个线程在操作

3. 子线程怎么更新界面？
   → 不能直接改，要用 Invoke 让主线程代劳
```

> **面试话术：** "在我的上位机项目中，我把通信采集放在子线程中运行，主线程专门负责UI响应。线程之间通过lock保证数据的线程安全，子线程需要更新界面时使用Invoke委托给主线程执行。这样既保证了数据采集的实时性，又保证了界面不会卡顿。"
